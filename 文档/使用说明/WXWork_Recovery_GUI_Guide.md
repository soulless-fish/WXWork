# 企业微信聊天恢复 GUI 使用说明

## 一、GUI 能做什么

这个 GUI 是对恢复脚本和整理脚本的桌面封装，适合日常重复操作。

正式相关文件：

- `wxwork_recovery_gui.py`
- `打包配置\构建脚本\Build_WXWork_Recovery_GUI_Exe.ps1`
- `dist\WXWorkRecoveryGUI.exe`

主要能力：

- 扫描本机 `Documents\WXWork` 企业账号目录
- 选择企业账号 ID 并匹配对应的 `WXWork.exe` 进程
- 直接执行聊天恢复，无需手敲完整命令
- 按秒级间隔自动重跑恢复，持续抓取当前缓存中的最新聊天消息
- 扫描最新恢复 sqlite 中的全部会话
- 默认先显示“聊天会话（内/外部）”，只看 `R:` 外部群和 `R:` 内部/部门群
- 如需看 `S:`、`Y:`、`F:`、`M:`、审批、邮件等其它类型，可切到“全部会话”或对应筛选
- 聊天整理页已改成“左侧会话列表 + 右侧消息预览”的查看方式
- 选中会话后，可以直接在右侧查看发送时间、发送人、消息摘要和命名依据
- 将单个会话或当前筛选结果批量整理导出
- 在 GUI 中直接查看会话名称、会话 ID、消息数、最后时间、对方ID和高频发送人
- 直接打开最近导出索引、全部会话总表、CSV、报告和输出目录
- 顶部已增加菜单栏，可直接进入“文件 / 聊天整理 / 帮助”

## 二、聊天整理页新布局

当前 `聊天整理导出` 页已不再只是一个会话表格，而是更接近第三方查看器：

- 顶部是整理输出目录、最新 sqlite 和常用打开入口
- 左侧是会话列表，主要看会话名、最近时间、消息数和类型
- 右侧上方是会话信息卡，显示：
  - 会话类型
  - 会话ID
  - 对方ID
  - 消息数 / 发送人数
  - 时间范围
  - 命名依据
  - 高频发送人
- 右侧下方是完整聊天内容浏览区，默认展示当前会话全部已恢复消息，并直接带出发送时间和发送人

## 三、标准使用顺序

1. 打开企业微信，并切到目标企业账号。
2. 打开目标聊天，尽量多上滑一些历史消息。
3. 启动 GUI。
4. 选择或输入企业账号 ID。
5. 点击 `扫描匹配进程`，必要时再点 `使用所选 PID`。
6. 点击 `开始恢复`。
7. 如果希望持续追最新消息，可在 `聊天整理导出` 区域设置间隔秒数后点击 `开启实时恢复`。
8. 查看恢复出的 `*_readable.csv` 和 `*_report.md`，或直接等待会话列表自动刷新。
9. 在 `聊天整理导出` 区域点击 `刷新会话列表`，或使用实时恢复自动刷新结果。
10. 在左侧会话列表中单击一个会话，先看右侧完整聊天内容是否已对上会话名、客户名和发送时间。
11. 视需要切换 `显示范围`，再选择 `整理所选会话` 或 `整理当前筛选结果`。
12. 打开最近导出的索引文件或进入对应会话目录查看结果。

## 四、在新电脑或商务电脑上使用

当前 GUI 已支持在新的 Windows 电脑上做离线导出，但前提必须满足下面几条：

1. 必须在那台电脑本机登录过目标企业微信账号。
2. 那台电脑本机必须存在 `Documents\WXWork` 本地目录。
3. 执行恢复时，目标账号对应的 `WXWork.exe` 必须正在运行。
4. 目标聊天窗口最好已经打开，并且已经上滑加载过更多历史消息。

如果只是把当前电脑上的 GUI 打开，并不能替商务电脑“隔空恢复”聊天记录；恢复能力始终依赖目标机器自己的本地企微缓存。

当前版本在新电脑上的默认输出目录逻辑是：

1. 如果仍在当前项目仓库里运行，则继续兼容当前仓库目录。
2. 如果换到新的电脑或新的目录运行，则默认输出到当前用户 `Documents\WXWorkRecovered`。
3. 如果该目录不可用，则回退到程序所在目录下的 `WXWorkRecovered`。

这样做的目的，是让 GUI EXE 脱离固定的 `<PROJECT_DIR>` 路径，也能直接离线导出。

## 五、实时恢复怎么用

如果你想让 GUI 持续追踪新消息，而不是每来一批消息都手工点一次 `开始恢复`，可以直接使用实时恢复：

1. 先保持企业微信打开，并停留在目标账号。
2. 尽量打开目标聊天窗口；如果还想补更多历史，先继续上滑消息。
3. 在 `聊天整理导出` 区域填写 `间隔秒数`，例如 `1`。
4. 点击 `开启实时恢复`。
5. GUI 会按间隔自动重跑恢复，并自动刷新：
   - 最新 sqlite
   - 会话列表
   - 当前选中的聊天内容预览
6. 不想继续时，点击 `停止实时恢复`。

注意：

- 这里的“实时”是指 GUI 自动定时重跑恢复，不是直接监听企业微信内部新消息事件。
- 当前已经允许把间隔设为 `1` 秒；如果单轮恢复本身就超过 `1` 秒，GUI 会在上一轮结束后立即补跑下一轮，不会并发叠加多轮恢复。
- 新消息只有在当前企业微信进程里已经被加载到缓存后，下一轮自动恢复才更容易抓到。
- 如果某个聊天窗口从未打开过，或者历史消息还没被滚动加载出来，恢复结果仍可能偏少。

## 六、整理导出后会得到什么

每个会话目录下现在默认会生成中文文件名：

- `时间线.csv`
  这个文件是该会话的全量消息明细表。每一行基本对应一条恢复出来的消息，适合做筛选、排序、导入 Excel。
- `发送人映射.csv`
  这个文件按发送人聚合，统计每个 `sender_id` 发了多少条消息，并尽力推测其姓名。适合先看“谁在说话”。
- `媒体索引.csv`
  这个文件只保留带媒体线索的消息，例如图片、媒体下载链接、文件/卡片等，适合单独追图片或文件。
- `会话时间线.md`
  这是更适合直接阅读的 Markdown 时间线。
- `会话摘要.json`
  这是给程序或技术排查用的摘要信息。
- `导出文件说明.txt`
  这是给普通用户快速解释目录内文件用途的说明。

整理输出根目录下还会生成：

- `当前筛选会话索引.csv`、`外部群索引.csv`、`单聊索引.csv`、`助手应用索引.csv` 等
  这些是本次导出的筛选结果索引。
- `全部会话总表.csv`
  这是当前恢复 sqlite 中扫描到的全部会话总表，不只限于本次导出的那一批。
- `recovery_manifest.json`
  这是批次级导出清单，供后续同步到另一个项目做知识清洗、结构化入库和知识库/RAG 接入时使用。

`recovery_manifest.json` 当前至少会记录：

- `source_machine`
- `source_operator`
- `enterprise_account_id`
- `recovery_batch_id`
- `exported_at`
- `session_count`
- `files`
- `sessions`

注意：

- `已选会话索引_*.csv` 这类文件只是索引，不是聊天明细本体。
- 真正的单个会话聊天记录，请进入对应会话目录后打开 `时间线.csv`。

## 七、会话名称说明

GUI 会话列表现在会同时显示：

- 左侧列表里主要显示：
  - `会话名称`
  - `最近时间`
  - `消息数`
  - `会话类型`
- 右侧信息卡里会补充显示：
  - `对方ID`
  - `会话ID`
  - `命名依据`
  - `高频发送人`

其中 `会话名称` 不是数据库里总能直接拿到的正式名称，而是脚本尽力推测的结果：

- 单聊现在会优先尝试读取本机 `Documents\WXWork\<当前账号>\Index\data_index.db` 里的联系人索引，因此能补回一部分真实联系人名或备注名。
- 如果单聊仍拿不到可信联系人名，脚本会优先退回 `单聊_<对方ID>`，避免继续显示误导性的正文片段。
- `R:` 会话并不一定全都是真正的外部群，当前 GUI 已把明显的内部/部门群单独拆出来显示。
- 外部群除了会从当前恢复消息里猜群名，也会尽量参考本机 `Index\data_index.db` 里的会话索引；例如缺失的门店群名有机会从本机索引补回。
- 如果群名不够可信，脚本会退回到更保守的命名方式，例如按活跃成员做 best-effort 标注，而不是硬猜一个错误群名。
- `发送人映射.csv` 里最稳定的标识通常是数字型 `sender_id`；`sender_name_guess` 只是根据当前恢复内容做 best-effort 推测。

为什么别人的恢复工具经常能把群名、客户名、时间对得更完整：

- 因为它通常不只依赖当前恢复出来的 `message_table_partial`。
- 更完整的工具往往还会结合本机 `Documents\WXWork\<账号>\Index\data_index.db` 之类的本地索引、联系人索引和会话索引。
- 本项目现在也已经接入这部分本机索引做补名，所以单聊和一部分门店群名称会比以前更准。
- 但如果某些名称在当前恢复数据和本机索引里都不存在，仍然只能 best-effort，不能承诺 100% 还原。

因此，实际判断时请同时参考 `会话名称`、`对方ID` 和 `会话ID`。

## 八、启动方式

Python 方式：

```powershell
python .\wxwork_recovery_gui.py
```

EXE 方式：

```powershell
.\dist\WXWorkRecoveryGUI.exe
```

重建 EXE：

```powershell
powershell -ExecutionPolicy Bypass -File .\打包配置\构建脚本\Build_WXWork_Recovery_GUI_Exe.ps1
```

如果需要把 GUI 交付到新的电脑上，优先建议直接带上：

1. `dist\WXWorkRecoveryGUI.exe`
2. 与之配套的说明文档
3. 或者直接在目标电脑上重新执行一次打包脚本

## 九、重要限制

GUI 并不会改变恢复原理。

它仍然依赖运行中的 `WXWork.exe` 进程里已经出现过的解密 SQLite 页面。

实时恢复模式也一样：它只是自动按间隔重新执行恢复，不代表企业微信新消息一到达就能被底层立即监听到。

如果恢复结果偏少，优先这样处理：

1. 回到企业微信。
2. 进入目标聊天继续上滑更多历史消息。
3. 再重新执行恢复和整理。
