# 项目详细链路说明（供聊天恢复 Codex 阅读）

本文基于 `2026-04-22` 当前仓库状态编写，目标是给另一个 Codex 快速建立统一认知，避免把“当前实时自动回复主链路”和“企业微信本地聊天恢复/知识库建设”混成一条线。

核心结论先写在前面：

1. 当前项目真正已经跑通的主链路，是 `Mac mini 本地服务器 + 1 个安卓模拟器 + 模拟器企微 + WorkTool WebSocket/HTTP + 本地后端 + 大模型生成 + WorkTool 回发`。
2. 企业微信本地聊天恢复，不是当前实时回复主链路，而是后续“历史知识沉淀、案例库、向量检索增强”的离线数据来源。
3. 另一个 Codex 如果现在只能在“我这台本机”恢复聊天记录，不代表它已经具备“恢复商务电脑聊天记录”的能力。因为恢复能力依赖的是“那台商务电脑自己的本地企微缓存和运行进程”，不能隔空替代。
4. 后续如果要把聊天记录接到本项目里，推荐优先做“离线知识库/RAG 检索增强”，而不是一上来就自己训练一个新模型。
5. 这个知识库方向不强制依赖 OpenClaw。当前项目已经支持 `openai_compatible`，可以直接通过 OpenAI 兼容中转或其他兼容模型服务继续扩展。

## 一、当前项目真实主链路是什么

这部分以 [README.md](/Users/apple/wecom-external-group-ai-cs/README.md)、[architecture.md](/Users/apple/wecom-external-group-ai-cs/docs/architecture.md)、后端实际代码为准。

### 1. 部署与运行形态

当前项目真实运行形态是：

1. `Mac mini` 运行后端和前端。
2. `Mac mini` 本机再运行 `1` 个安卓模拟器。
3. 模拟器里安装企业微信和 `WorkTool`。
4. 使用一个专门值守外部群的企微账号登录。
5. `WorkTool` 把群消息通过本地 `WebSocket/HTTP` 推给后端。
6. 后端识别消息来源，只有被判定为 `external_user` 的消息才会进入 AI 自动回复。
7. 后端调用当前配置的大模型生成回复草稿。
8. 后端再通过 `WorkTool` 本地长连接或官方 WorkTool API 把回复送回去，由模拟器里的企微执行发送。

### 3. 当前实时自动回复链路的标准顺序

标准顺序可以理解成下面这条：

```text
外部群真实消息
-> 模拟器中的企业微信
-> WorkTool WebSocket / HTTP
-> worktool_adapter.py 归一化
-> external_group_auto_reply.py
-> group_messages 入库
-> AIProvider.draft_reply(...)
-> reply_tasks / 本地下发队列
-> WorkTool 或官方 WorkTool API
-> 模拟器企业微信执行发送
```

### 4. 当前项目已经支持的消息来源，不只一条

当前代码里，外部群消息来源实际已经支持三类：

1. `WorkTool` 本地消息桥接
2. 第三方桥接器 / SCRM 通过 `source_push` 推送归一化消息
3. 企业微信官方会话存档 SDK 拉取后再送入自动回复链路

但要注意：

1. 这三类来源都属于“在线消息接入层”。
2. “本地聊天恢复 GUI 导出的 CSV/Markdown/JSON”不属于当前已实现的在线消息入口。
3. 本地聊天恢复导出的数据，应该被视为“后续离线知识数据源”，而不是直接塞进当前实时收消息入口。

## 二、当前项目并不强制依赖 OpenClaw

这部分请直接以 [backend/app/core/config.py](/Users/apple/wecom-external-group-ai-cs/backend/app/core/config.py) 和 [backend/app/services/ai/factory.py](/Users/apple/wecom-external-group-ai-cs/backend/app/services/ai/factory.py) 为准。

当前 AI Provider 已支持：

1. `openai_compatible`
2. `deepseek`
3. `zhipu`
4. `openclaw`
5. `mock`

并且默认值就是：

1. `AI_PROVIDER=openai_compatible`

这意味着：

1. 当前项目后续如果接“中转 OpenAI 兼容接口”，是完全可行的。
2. 后续如果做知识库/RAG，也不需要先把 OpenClaw 接进来才能做。
3. OpenClaw 更像是一个可选的 Agent/CLI 包装层，不是这个项目知识库化的唯一前提。

更准确的说法应该是：

1. `知识检索层` 负责找材料。
2. `生成模型层` 负责组织回复。
3. 这个生成模型层可以是 `openai_compatible` 直连，也可以是 `OpenClawProvider`。

## 三、企业微信本地聊天恢复和当前项目是什么关系

企业微信本地聊天恢复，不是当前项目实时客服的消息入口，而是未来知识建设的“离线历史数据补充链路”。

推荐把它理解成下面这条线：

```text
Windows 本机企业微信本地缓存
-> 恢复 GUI / 恢复脚本
-> 导出时间线.csv / 会话时间线.md / 会话摘要.json
-> 数据清洗 / 分类 / 结构化
-> 知识库 / 向量库
-> 检索结果拼成 faq_context
-> 接回当前 AIProvider.draft_reply(...)
```

也就是说：

1. 当前项目主链路解决的是“客户现在发来一句话，系统怎么回”。
2. 本地聊天恢复解决的是“如何把历史真实聊天沉淀成可检索的知识和案例”。
3. 这两条能力最终会在 `faq_context` 这个注入点汇合，而不是直接互相替代。

## 四、为什么“本机能恢复”不等于“商务电脑也能恢复”

这部分请严格参考 [WXWork_Recovery_GUI_Guide.md](/Users/apple/wecom-external-group-ai-cs/WXWork_Recovery_GUI_Guide.md)。

恢复 GUI 的关键限制是：

1. 它依赖运行中的 `WXWork.exe` 进程。
2. 它依赖当前那台机器自己的 `Documents\\WXWork` 本地目录。
3. 它只能恢复“已经被当前企业微信进程加载到本地缓存/SQLite 页面里的数据”。
4. 如果某个聊天窗口从未在这台机器上打开过，或者历史消息没上滑加载出来，恢复结果就会偏少。

因此必须明确：

1. 另一个 Codex 如果只在“当前这台电脑”上运行恢复工具，它拿到的只是“这台电脑”的本地企微缓存。
2. 它不能替代“商务电脑”的本地企微缓存。
3. 想拿商务电脑上的聊天记录，必须在商务电脑本机运行同类恢复流程，或者通过远程桌面进入商务电脑后，在商务电脑自身环境执行。

这件事的本质不是“路径没配好”，而是“恢复原理依赖机器本地缓存”。

## 五、商务电脑要如何接入这个恢复能力

正确目标不是“在我当前电脑上替商务电脑恢复”，而是：

1. 把恢复 GUI 或恢复脚本部署到商务电脑。
2. 在商务电脑本机登录目标企业微信账号。
3. 打开要恢复的会话，并上滑加载更多历史消息。
4. 在商务电脑本机执行恢复 GUI。
5. 在商务电脑本机导出标准化结果。
6. 再把这些导出结果同步回当前项目可读的位置，用于后续知识清洗和入库。

推荐的商务电脑落地步骤如下：

1. 在商务电脑准备恢复工具的可运行版本，最好是：
   `Python 脚本 + 依赖环境`
   或
   `PyInstaller 打包后的 EXE`
2. 在商务电脑确认能看到：
   `Documents\\WXWork`
3. 在商务电脑确认目标企业微信账号已经登录，而且 `WXWork.exe` 正在运行。
4. 在商务电脑打开目标聊天窗口，并尽可能上滑更多历史消息。
5. 在商务电脑运行恢复 GUI，选择企业账号 ID，匹配进程 PID，执行恢复。
6. 在商务电脑导出：
   `时间线.csv`
   `发送人映射.csv`
   `媒体索引.csv`
   `会话时间线.md`
   `会话摘要.json`
7. 统一把导出结果放入一个稳定目录，例如：
   `D:\\WXWorkRecovered\\<机器名>\\<日期批次>\\`
8. 再通过共享目录、同步盘、Git 仓库、手工复制或后续专门的上传脚本，把这批导出文件同步到本项目侧。

## 六、聊天记录导出后，应该怎样和当前项目结合

这里最重要的一条原则是：

1. 不要把“本地恢复出来的历史聊天”直接混写进当前 `group_messages` 作为实时运行日志。

原因是：

1. [backend/app/models/group.py](/Users/apple/wecom-external-group-ai-cs/backend/app/models/group.py) 里的 `group_messages` 现在承担的是“在线消息接入、超时兜底、实时回复触发、时间线排障”的运行职责。
2. 历史恢复数据和实时接入数据，时间语义、来源可信度、去重规则、补录方式都不一样。
3. 如果把两类数据硬塞在一起，后续很容易污染自动回复触发逻辑、超时任务逻辑和排障判断。

因此更合理的结合方式是：

1. 恢复原始数据先进入“离线恢复原始层”。
2. 清洗后的结构化消息进入“离线消息明细层”。
3. 再从结构化消息中抽取 FAQ、案例、话术、规则、风险提示，进入“知识层/向量层”。
4. 最后只把“检索出来的高价值上下文”注入当前自动回复链路。

## 七、推荐的数据分层方案

### 1. 原始恢复层

职责：

1. 保存恢复 GUI 导出的原始文件和原始记录。
2. 保留批次、来源机器、企业账号、导出时间、原始文件路径。
3. 这一层不做过强语义判断，只保证“可追溯、可重跑、可回放”。

建议至少记录这些字段：

1. `source_machine`
2. `source_operator`
3. `enterprise_account_id`
4. `recovery_batch_id`
5. `session_id`
6. `session_type`
7. `session_name`
8. `peer_id`
9. `message_time`
10. `sender_id`
11. `sender_name`
12. `raw_text`
13. `raw_message_type`
14. `raw_file_path`
15. `raw_json`

### 2. 清洗结构化层

职责：

1. 把不同来源的 CSV/Markdown/JSON 统一成项目自己的标准消息结构。
2. 补齐统一时间格式、统一会话键、统一发送者类型。
3. 过滤明显无效数据。

这一层建议完成这些处理：

1. 时间统一到 `Asia/Shanghai` 或统一 UTC 存储。
2. 文本规范化，去掉多余空白、控制字符、无意义换行。
3. 去重，建议按：
   `企业账号 + 会话ID + 发送人 + 时间戳 + 归一化文本`
   做近似去重。
4. 会话分类，至少区分：
   `外部群`
   `企业内部群`
   `单聊`
   `助手应用/系统消息`
5. 发送者分类，至少区分：
   `external_user`
   `staff`
   `unknown`
6. 消息分类，至少区分：
   `text`
   `image`
   `file`
   `link`
   `system`
7. 脱敏，必要时对手机号、身份证、地址、订单号做脱敏副本。

### 3. 知识抽取层

这一步不是简单把所有聊天都丢给向量库，而是把“有业务价值、能服务自动回复”的内容抽出来。

优先抽这几类：

1. 高频问答
2. 标准解释话术
3. 售前/售后/税务/营业执照/维修流程等业务专题案例
4. 明确的禁答项、风险项、升级人工条件
5. 成功成交案例与失败案例
6. 针对不同群、不同门店、不同业务线的专属话术

不建议直接纳入正式知识库的内容：

1. 纯闲聊
2. 情绪性争吵但没有知识价值的内容
3. 明显过时的政策说法
4. 无法确认真实性的内部猜测
5. 带强隐私且没有必要保留原文的内容

### 4. 向量切片层

推荐不要按“整段长聊天”直接做向量，而是做适度切片。

比较稳妥的切片方式：

1. 一问一答对切片
2. 同一主题下连续多轮对话切成一个案例块
3. 群公告/标准说明切成规则块
4. 人工总结出的标准 SOP 切成知识块

每个切片建议至少带这些元数据：

1. `knowledge_id`
2. `chunk_id`
3. `source_type`
4. `source_machine`
5. `source_session_id`
6. `source_message_range`
7. `business_topic`
8. `applicable_group_scope`
9. `applicable_store_scope`
10. `speaker_role`
11. `updated_at`
12. `quality_score`
13. `review_status`

## 八、向量化与入库建议

这里更推荐先做“向量检索增强”，不是先训练一个自己的新模型。

### 1. 为什么不建议先训练新模型

原因很实际：

1. 当前阶段你最缺的是“可检索、可复用、可治理的高质量语料”。
2. 不是“缺一个自己训练的大模型”。
3. 先把历史聊天做成可检索知识库，收益更快，也更容易验证是否真的提升回复质量。

### 2. 推荐能力拆分

推荐拆成两种模型能力：

1. `Embedding/向量模型`
   负责把知识片段向量化，用于检索召回。
2. `生成模型`
   负责根据客户问题和检索结果组织最终回复。

### 3. 当前项目最适合的接法

当前项目已经有 `faq_context` 这个注入点：

1. [backend/app/services/ai/base.py](/Users/apple/wecom-external-group-ai-cs/backend/app/services/ai/base.py)
2. [backend/app/services/wecom/external_group_auto_reply.py](/Users/apple/wecom-external-group-ai-cs/backend/app/services/wecom/external_group_auto_reply.py)
3. [backend/app/api/routes/replies.py](/Users/apple/wecom-external-group-ai-cs/backend/app/api/routes/replies.py)

所以最小改造路径应该是：

```text
客户消息
-> 检索服务按当前群/业务线召回 TopK 知识片段
-> 把召回结果拼成 payload["context"] / faq_context
-> 继续调用 AIProvider.draft_reply(...)
-> 生成回复草稿
-> 继续走现有 reply_tasks / WorkTool 发送链路
```

也就是说，知识库最优先接入点不是改掉整个自动回复服务，而是先在 AI 调用前增加一个“检索增强层”。

### 4. 向量库存储建议

当前项目实时运行库主要还是围绕业务表设计，不建议把大规模向量长期堆在当前 SQLite 里。

更稳妥的方向是：

1. 优先考虑 `PostgreSQL + pgvector`
2. 或单独使用 `Qdrant / Milvus / Weaviate` 这类向量库
3. 小规模本地验证时可以先做轻量实现，但不要把它当最终长期架构

## 九、推荐的离线导入到在线服务的完整流程

推荐最终落成下面这条链路：

```text
商务电脑 / 本机企微
-> 本机恢复 GUI 导出
-> 原始恢复层
-> 清洗结构化层
-> 知识抽取层
-> 人工抽检 / 审核
-> 向量化
-> 向量库
-> 在线检索服务
-> 生成 faq_context
-> ExternalGroupAutoReplyService
-> AIProvider.draft_reply(...)
-> reply_tasks
-> WorkTool / 官方 WorkTool API
-> 外部群回发
```

### 这条链路里哪些是当前已实现的

当前已经实现的是：

1. `ExternalGroupAutoReplyService`
2. `AIProvider.draft_reply(...)`
3. `reply_tasks`
4. `WorkTool` 本地发送和官方 WorkTool API 发送
5. 在线消息的入库和规则判断

### 这条链路里哪些是后续要新增的

后续需要新增的是：

1. 本地恢复导入器
2. 清洗与结构化任务
3. 知识抽取与审核流程
4. 向量化任务
5. 向量检索服务
6. 检索结果注入 `faq_context` 的统一适配层

## 十、是否一定需要“Agent 能力模型”

不一定。

对当前项目的近期目标来说，优先级更高的是：

1. 有一个稳定的生成模型
2. 有一个稳定的向量检索层
3. 有一套可治理的知识数据流

在以下场景里，普通 `RAG + 生成模型` 往往就够了：

1. 客户问标准问题
2. 需要参考历史案例生成回复
3. 需要根据当前群专属客服 prompt 生成话术

只有在以下场景里，Agent 能力才明显更有价值：

1. 需要多步推理
2. 需要跨多个知识源/系统自动查数据
3. 需要自动调用外部工具
4. 需要自动创建工单、查询订单、联动 CRM

所以对这个项目的建议是：

1. 第一阶段先做 `恢复导出 -> 清洗 -> 知识库 -> 检索增强 -> 当前回复链路`
2. Agent 不是第一阶段前置条件
3. OpenClaw 也不是第一阶段前置条件

## 十一、另一个 Codex 现在最容易犯的误区

请特别避免下面这些误区：

1. 误以为“恢复工具在我当前电脑能跑”，就等于“已经支持商务电脑聊天恢复”。
2. 误以为“本地聊天恢复”就是“当前实时自动回复主链路”。
3. 误以为“做知识库”就必须先接 OpenClaw。
4. 误以为“恢复出来的历史消息”应该直接写入当前 `group_messages`。
5. 误以为“只要把所有聊天都做向量化”就能自动得到高质量知识库。

你必须先接受一个前提：

2. 把恢复能力整理成“商务电脑可直接使用”的交付形式，优先选择：
   `可直接运行的 EXE`
   或
   `带完整依赖说明的 Python 运行包`
3. 确保商务电脑版本支持：
   自动扫描 `Documents\\WXWork`
   选择企业账号 ID
   匹配 `WXWork.exe` 进程
   触发恢复
   导出标准化结果文件
4. 明确要求用户在商务电脑上先打开目标聊天窗口，并上滑更多历史消息，再执行恢复；否则恢复结果会不完整。
5. 给商务电脑恢复结果定义统一输出目录和统一文件结构，至少保留：
   `时间线.csv`
   `发送人映射.csv`
   `媒体索引.csv`
   `会话时间线.md`
   `会话摘要.json`
6. 额外生成一份批次级别的清单文件，例如：
   `recovery_manifest.json`
   里面至少写明：
   `source_machine`
   `enterprise_account_id`
   `recovery_batch_id`
   `exported_at`
   `session_count`
   `files`
7. 不要尝试把恢复出来的历史消息直接写进当前项目的 `group_messages` 表。
8. 你的产出重点是：
   让商务电脑可以稳定导出聊天记录
   让导出结果具备后续清洗/知识入库价值
9. 如果你要继续往当前项目结合，下一步应该做的是：
   `恢复导出标准化`
   `清洗结构化`
   `知识抽取`
   `向量化`
   `检索结果注入 faq_context`
   而不是直接改实时自动回复主链路。

你的最终交付至少要包含：

1. 商务电脑部署/运行说明
2. 商务电脑恢复工具的可执行形式
3. 标准导出目录结构
4. 批次清单文件格式
5. 已知限制说明
6. 后续如何把导出结果喂给当前项目知识库链路的建议

## 十三、推荐的后续实施顺序

如果要把这件事继续推进成正式能力，推荐顺序如下：

1. 先把商务电脑恢复能力做稳定。
2. 再统一恢复导出格式和批次清单。
3. 再做离线清洗与结构化入库。
4. 再做知识抽取和人工审核。
5. 再做向量检索服务。
6. 最后把检索结果接入当前 `faq_context` 注入点。

这样推进的好处是：

1. 不会误伤当前已经能跑的实时自动回复主链路。
2. 每一步都能独立验证。
3. 后续无论生成层继续用 OpenAI 兼容接口还是 OpenClaw，都可以复用同一套知识底座。
